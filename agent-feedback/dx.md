## Let the Netlify preview fixtures run offline (cache the edge runtime or skip when it can't be fetched)

`packages/run/src/__tests__/main.test.ts` › `testPage` | 2026-07-23 | impact:med | effort:med

The `netlify-adapter-edge` and `netlify-adapter-not-edge` preview fixtures shell out to the Netlify CLI, which downloads its edge-functions (Deno) runtime on the first `preview` run. In a restricted/offline environment that download fails (`Download failed with status code 403` → `Netlify CLI has terminated unexpectedly` / `Error: fetch failed`), so those fixtures cannot run and there is no offline fallback. (An earlier, broader "dozens of fixtures cascade" symptom was actually a separate adapter-resolution bug — adapter-less fixtures defaulting to the Netlify adapter — now fixed by declaring the adapters at the workspace-root `package.json`; only these two explicit Netlify fixtures still need the download.) `testPage` already closes its server in `finally`, so the request here is not another `server.close()`: cache/vendor the edge runtime for CI, or mark these fixtures `skip_preview` (with a warning) when the runtime is unavailable, so a blocked download is one skipped fixture instead of a failure that blocks running the rest of the suite locally.

## Add a canonical-origin option to the static adapter so build-time absolute URLs are not http://localhost

`packages/adapters/static/src/crawler.ts` › `createCrawler` | 2026-07-18 | impact:med | effort:low

The static crawler falls back to `const origin = opts.origin || "http://localhost"` (crawler.ts:22) and builds every request as `new URL(path, origin)` (crawler.ts:46), so `ctx.url.origin` inside handlers and pages is `http://localhost` at build time. For the default-entry (in-process) path, `buildEnd` constructs the crawler with only `{ out, notFoundPath }` and never passes an `origin` (index.ts:150-161); the spawn-server path is no better, using `http://localhost:${randomPort}` (index.ts:165, 193). The public `Options` interface exposes only `urls` (index.ts:25-50), so there is no hook to set the site origin. This breaks any build-time absolute URL: RSS `<link>`/`<guid>`, sitemap `<loc>`, and canonical/OG tags all render under `http://localhost` (RSS and sitemap specs require absolute URLs, so this blocks correct feeds/sitemaps). The only workaround is to hardcode/env-var the origin in app code. Fix: expose an `origin`/`site` option on `Options` and thread it into `createCrawler`'s `opts.origin` (and the spawn-server `origin`). Astro's `site` and SvelteKit's paths config are precedent. Docs (website/docs/marko-run/adapters.md, Static section) currently document only `urls`.

## Add a cookie/session helper to the `@marko/run` Context (or at least document the recipe)

`packages/run/src/runtime/types.ts` › `Context` | 2026-07-18 | impact:med | effort:med

The `Context` interface (`packages/run/src/runtime/types.ts:971-995`) and its constructor `createContext` (`packages/run/src/runtime/internal.ts:181-277`) expose `request`, `url`, `body`, `data`, `redirect`, `back`, `fetch`, and `render`, but nothing for cookies. Reading a session in a root `+middleware.ts` therefore requires `context.request.headers.get("cookie")` plus a hand-written parser, and every `+handler.ts` that persists state (cart, auth, flash messages) must build a `Set-Cookie` string by hand and `response.headers.append("Set-Cookie", ...)`. This is the framework's own headline example: the docs invoke "reading a session" / "the current session" as the motivating middleware and data-loading use case (data-loading.md:3, :23; typescript.md:30) yet never show the cookie plumbing a session requires, and there is no cookie example anywhere in the @marko/run docs. The only cookie handling in the repo is the `basic-cookies` test fixture doing manual header string-parsing, so even the test suite has no helper. The primitive already exists internally but is unsurfaced: the dev server hand-parses request cookies for its client id (`packages/run/src/adapter/dev-server.ts:80-89`) and the node adapter forwards response `Set-Cookie` via `getSetCookie()` (`packages/run/src/adapter/middleware.ts:94-96`). Provide a first-class `ctx.cookies` (get/set/delete with attributes) as peer WHATWG meta-frameworks do, or at minimum a documented cookie/session recipe in data-loading.md/runtime.md. Manual handling does work (an HttpOnly `Set-Cookie` on a 302 round-trips, `Max-Age=0` clears), so if the WHATWG-minimal design is intentional the low-effort fix is purely the documented recipe.

## Type `context.search`/`$global.search` as the parsed record, not `undefined`, when no search validator is declared

`packages/run/src/runtime/types.ts` › `RouteDef` | 2026-07-18 | impact:med | effort:low

Reading any query param without first declaring a `search` validator is a type error even though it works at runtime. `RouteDef.search` (`packages/run/src/runtime/types.ts:238-246`) and `RouteForFileDef.search` (`:531-535`) resolve to `undefined` when the merged options contain no `search` key, so `ctx.search.fail` reports TS18048 "'ctx.search' is possibly 'undefined'" and a `never` probe shows the field is typed exactly `undefined`. But the runtime getter always returns an object: `createContext`'s `get search()` computes `searchParamsToObject(url.searchParams)` and passes it through a validator only when one exists (`route?.options.search ? route.options.search(search) : search`, `internal.ts:217-228`), never `undefined`. The type therefore contradicts the runtime, and is asymmetric with the sibling `params` field, which falls back to the raw `PathParams<Path>` instead (`types.ts:229-237`, `:526-530`). Without a validator, `search` should type as the raw parsed record (mirroring `params`, e.g. `Record<string, string | string[]>`); at minimum the "must declare a search validator to read search" requirement should be documented. The same asymmetry hits `$global.search` in templates, and run's own route-options fixture and the README's `context.search.page` example only work because a `+middleware.ts` happens to supply a validator, with no note that omitting it makes the field `undefined`.

## Provide (or document) a client-side URL/navigation primitive; `$global.url` is a frozen SSR snapshot

`packages/run/src/runtime/client.ts` › `window.Run` | 2026-07-18 | impact:med | effort:med

The entire client-facing @marko/run runtime surface is `window.Run.href`, a compile-time URL-string builder assigned by `client.ts:3` (`(window as any).Run ||= { href }`). There is no client navigation, no `history.pushState` wrapper, and no `popstate` helper anywhere in `packages/run/src/runtime`, `.../vite`, or the adapters. `$global` is the request `Context` whose `url` is a plain `URL` built once per request (`internal.ts:185`) and serialized to the browser as a literal `new URL("http://.../?page=1")` constant with no subscription to `window.location`. So for the mainstream archetype (a query-param-driven sort/filter/pagination/infinite-scroll table with shareable URLs), after a client `history.pushState` the reactive `$global.url` never updates, and browser back/forward silently desyncs the UI from the address bar unless the developer hand-writes a `popstate` listener that reads `window.location` rather than the reactive URL. Meanwhile runtime.md:24 states "$global.params and $global.url are serialized to the browser by default, so client-side code can read them after hydration", which reads as if that snapshot were a live source of truth. Provide a small client primitive (e.g. a `Run.navigate` / URL-sync helper that also refreshes the reactive URL source) or, at minimum, document that it is an immutable SSR snapshot, that current client state comes from `window.location`, and show the pushState/popstate pattern.

## Warn (or fail) when `marko-run dev`'s requested port is in use instead of silently binding a random ephemeral port

`packages/run/src/vite/utils/server.ts` › `getAvailablePort` | 2026-07-18 | impact:low | effort:low

`getAvailablePort` (`packages/run/src/vite/utils/server.ts:238-249`) returns the requested port only when free: `if (port && !(await isPortInUse(port))) return port;`, otherwise it falls through to `net.createServer().listen(0)`, which yields an OS-assigned ephemeral high port. The dev CLI resolves the port as `port ?? server.port ?? preview.port ?? defaultPort` where `defaultPort = Number(process.env.PORT || 3000)` (`cli/commands.ts:111-119`, `vite/plugin.ts:70`) and passes it through this function, so starting a second app while `PORT=3225` is taken does not error and does not increment to 3226 — it jumps to a random port. No "port in use / trying another" notice is printed anywhere (a grep of the runtime for `in use|EADDRINUSE|another port` finds none), unlike Vite/Next which warn and increment sequentially. The final bound port does appear in the startup banner so it is not fully silent, keeping impact low, but a developer who configured a proxy or shared a `localhost:$PORT` URL is silently pointed elsewhere. Either fail loudly or increment predictably (3225 -> 3226), and in all cases print that the requested port was unavailable and which was chosen. While here, the CLI help (`cli/index.ts:57-59`) says dev `--port` defaults to `preview.port` then `$PORT`, but `commands.ts:113-116` prefers `server.port` first — a minor help/impl wording mismatch worth aligning at the same time.

## Warn when a `+meta.*` file sits in a directory with no `+page`/`+handler` — it is silently dropped and never inherited

`packages/run/src/vite/routes/builder.ts` › `traverse` | 2026-07-18 | impact:med | effort:low

A `+meta.*` placed in a page-less directory (e.g. a section shell holding only a `+layout.marko` plus child page dirs) produces no route metadata, no `D` type in the emitted `routes.d.ts`, and no diagnostic — the natural "section-level meta next to a section layout" expectation for a docs site fails silently. Layouts and middleware accumulate down the tree into `currentLayouts`/`currentMiddleware` (`builder.ts:194-209`) and apply to every descendant (`layouts: page ? [...currentLayouts] : []`, `:268`), but meta has no such accumulation: it is read only as `meta: dir.files.get(RoutableFileTypes.Meta)` at `builder.ts:269`, _inside_ the `if (page || handler)` block (`:232`), so a directory that holds only a layout and child dirs never reads its own `+meta` at all. That single leaf meta then flows through codegen unchanged (`routeFileIter` yields only `route.meta`, `codegen/index.ts:806`). No diagnostic catches it either: `warnNonRoutableLookalike` (`builder.ts:40`) only fires for names that fail `RoutableFileRegex`, and `+meta.json` matches, so it is deemed routable and never flagged; meta files are also never added to the `unusedFiles` set (see the companion cleanup finding). The minimal high-value fix is to warn on a `+meta` in a directory that yields no route; a larger enhancement would cascade and merge parent metas down the tree (the runtime type system already models meta as a mergeable tuple via `MergeTuple<NormalizedMetaFiles<...>>` at `runtime/types.ts`, hinting multi-file merge was anticipated). The docs are silent too: `website/docs/marko-run/file-based-routing.md` says meta "attach[es] to the route" but never states that it does not cascade nor that a page-less `+meta` is discarded.

## Adding or removing one route full-reloads every route module and re-emits the worker entry dozens of times

`packages/run/src/vite/plugin.ts` › `configureServer` | 2026-07-18 | impact:low | effort:med

Adding or removing a single leaf route triggers a full page reload of every existing route module plus a flood of worker-entry HMR updates, instead of invalidating just the route table and the one changed route. In the watcher handler (`plugin.ts:602-639`), when the changed routable file has no importers in the module graph — always true for a freshly added file, and for an unlinked module already dropped — the else branch loops over every key in `virtualFiles` and calls `devServer.watcher.emit("change", file)` for each non-`.marko` virtual file (`:630-636`), synthetically invalidating the entire set. Because the route table is a single shared virtual graph, that (and the importers branch at `:626-629`) propagates a page reload to every route module; observed as one `[vite] (client) page reload .marko-run/<route>.marko` per route for all ~14 routes on a single add. With the Cloudflare adapter each synthetic change also re-pushes `virtual:cloudflare/worker-entry`, producing dozens of duplicate `hmr update`/`hot updated` lines. The route-module reload cascade is run-attributable and adapter-independent; the worker-entry flood is amplified by `@cloudflare/vite-plugin`. Scope the emitted invalidation to the modules actually affected by the route delta rather than every non-`.marko` virtual file. This lives in the same watcher handler as the delete-wedges-dev-server bug but is a distinct symptom with a distinct fix.

The same watcher else-branch (`plugin.ts:630-636`) has a second, opposite failure mode, filed as the bug "Invalidate the `.marko` server-entries when a `+layout` is added": its `!file.endsWith(".marko")` guard SKIPS the `.marko` server-entries, so adding a `+layout` to an existing directory is silently not applied until an unrelated content edit invalidates that `.marko`. Any fix that narrows this invalidation to cut reload noise must still invalidate the affected `.marko` server-entries, or it will worsen the layout-drop correctness bug.

## Wire the generated `.marko-run/routes.d.ts` into type-checking so a fresh TS scaffold checks route files out of the box

`packages/run/src/vite/plugin.ts` › `writeTypesFile` | 2026-07-18 | impact:med | effort:med

A `-t basic` (TypeScript) scaffold cannot type-check its own route files: adding any `+handler.ts`/`+middleware.ts` that uses the global `Run` and running `npx tsc --noEmit` reports `TS2304: Cannot find name 'Run'` plus implicit-`any` on every `ctx`. The vite plugin generates the ambient/augmentation-bearing types via `writeTypesFile` (`plugin.ts:172`) into `.marko-run/routes.d.ts` (`typesDir = path.join(root, ".marko-run")`, `plugin.ts:465`; path built at `:180`) — outside `src/` — but never wires that path into compilation. The generated file carries `declare module "@marko/run"` (`packages/run/src/vite/codegen/index.ts:828`) and per-route-file `declare module "${info.modulePath}"` augmentations (`codegen/index.ts:947`); those only take effect when `routes.d.ts` is in the TS program, which requires it in the tsconfig `include`. The shipped template tsconfig uses `include: ["src/**/*"]`, which excludes it; adding `.marko-run/**/*` makes `tsc` pass with zero errors and yields precise typed data flow (`ctx.data.user` narrows to `User | undefined`). Docs already concede this manual step (`website/docs/marko-run/typescript.md:37`, "TypeScript will not include this file by default"), so it is a known-but-unshipped gap. The durable fix spans two places: emit an extendable tsconfig (as SvelteKit does with `.svelte-kit/tsconfig.json`) that projects `extends`, or a `src`-level ambient file with a `/// <reference>` to the generated types, plus fix the create-marko `basic`/`app` template tsconfig to include the generated path. Note the template tsconfig lives in create-marko (not this repo), but run owns the types location and the codegen, so the auto-wiring fix belongs here. Related to the missing types-only command below (no way to (re)generate these types without a full build).

## Add a types-only `marko-run typegen` command so `mtc`/`lint` don't check a stale `.marko-run/routes.d.ts`

`packages/run/src/vite/plugin.ts` › `renderVirtualFiles` | 2026-07-18 | impact:med | effort:med

`npm run lint` (which runs `mtc`) type-checks against whatever `.marko-run/routes.d.ts` the last dev/build emitted, not against current routes. `writeTypesFile` (`plugin.ts:172`) is only reached from `renderVirtualFiles` at `plugin.ts:375`, which runs exclusively during `dev`/`build`/`preview`; the CLI (`packages/run/src/cli/index.ts`) exposes only `preview`, `dev`, and `build` (`cli/commands.ts` `build()` at `:155`) — there is no types-only regeneration step. So deleting a `+meta.json` (or adding a handler/meta) and running `mtc` cold, with the dev server stopped, checks stale generated types: the route union in `routes.d.ts` still references the deleted file, `mtc` exits 0, and the error (or the already-fixed non-error) only surfaces after a fresh `npm run build`. The fix is a lightweight `marko-run typegen`/`prepare` subcommand that runs route discovery + `writeTypesFile` without bundling (the same fast path a type-check needs), or having the lint step trigger route regeneration first. This is the missing command that finding "Make `marko-run build` reject unknown flags" shows an app author already tried to fake via `--skip-bundle`, and its absence is also what makes the "misleading `unknown` cascade" below possible on a clean checkout.

## Make `marko-run build` reject unknown flags (sade `.strict()`); `--skip-bundle` is silently swallowed

`packages/run/src/cli/index.ts` › `prog` | 2026-07-18 | impact:med | effort:low

`marko-run build --skip-bundle` accepts the flag and does nothing, still emitting a full client+SSR bundle. The `build` command is declared with only `.option("-o, --output")` (`cli/index.ts:71-82`) and its action forwards just `opts.output/config/env`; sade (`sade("marko-run")`, `cli/index.ts:14`) is not `.strict()`, so any unknown flag is parsed into `opts` and ignored. `build()` (`packages/run/src/cli/commands.ts:155`) has no skip parameter and unconditionally runs both `viteBuild` passes — SSR (`commands.ts:213`) and client (`:228`). `grep -rn skip-bundle packages/run/src` returns nothing; the flag exists only in an app's own `typecheck` script (`marko-run build --skip-bundle && mtc`), where the author clearly wanted a bundle-free way to generate route types and the framework offers none, so every `npm run typecheck` pays the full bundle cost for output `mtc` never reads. Two complementary fixes: call `.strict()` so the bad flag is surfaced instead of silently accepted, and add a real types-only command (see "Add a types-only `marko-run typegen` command") so the fast path the app wanted actually exists. If `--skip-bundle` is intended to be supported, `build` must declare it and short-circuit after route-type generation.

## Degrade gracefully (or emit one clear diagnostic) when route types are missing during type-check, instead of a cascade of `unknown`-not-assignable errors

`packages/run/src/runtime/types.ts` › `Route` | 2026-07-18 | impact:med | effort:med

Type-checking `.marko` routes before `.marko-run/routes.d.ts` has been generated produces a wall of misleading errors on correct code, blaming the user's layout/handler rather than the missing codegen. Both `.marko-run/` and `*.tsbuildinfo` are typically gitignored, so a clean checkout has neither the generated types nor the tsc cache; running `npx mtc` (or opening the editor's TS server) before any `marko-run build`/`dev` triggers it. The route `data`/`$global.data` type flows through the per-file `Run` namespace augmentation that lives only in the generated file; without it, `data` falls back at `packages/run/src/runtime/types.ts:290-292` (`Data extends [infer T extends Record<string, unknown>] ? T : Record<string, unknown>`), so `data.player` is `unknown` and every downstream call expecting a concrete type (e.g. `TS2345 Argument of type 'unknown' is not assignable to parameter of type 'Player'`) fails with zero indication the real cause is ungenerated types. A subtle compounding hazard: with a stale `dist/.tsbuildinfo` present but `.marko-run` deleted, `mtc` can exit 0 off cached types, so whether route-shape drift is even caught depends on cache state. Better: when the augmentation is absent, degrade `data`/`search`/etc. to a documented `any` placeholder and/or surface a single diagnostic pointing at the missing codegen, and pair it with the discoverable `marko-run typegen` command above so the pre-step is real. Diagnostic quality for genuine `.marko` type errors is otherwise excellent; this is specifically the missing/stale-types failure mode. Directly connected to the missing types-only command and the swallowed `--skip-bundle` flag above.

## Wrap the `marko-run` CLI actions in try/catch so a build failure doesn't print a `triggerUncaughtException` stack

`packages/run/src/cli/index.ts` › `prog` | 2026-07-19 | impact:med | effort:low

Every failed `marko-run build` prints Vite's friendly error box and THEN dumps `node:internal/process/promises:394 triggerUncaughtException(err, true /* fromPromise */)` followed by a ~15-frame internal V8/rolldown/vite stack ending in `at async build (@marko/run/dist/cli/index.mjs)`, so the real error (e.g. "Duplicate routes", a malformed route dir, or a `+meta.json` trailing comma) is buried between two blocks of framework noise and reads like marko-run itself crashed. The `build`/`dev`/`preview` command actions are `async` arrow functions that `await build(...)` (`cli/index.ts:78-82`; dev at `:62-68`) and are dispatched by `prog.parse(process.argv)` (`cli/index.ts:84`); sade does not await or `.catch` the promise an action returns, and none of the actions wrap their body in try/catch. When either `viteBuild` pass rejects (`cli/commands.ts:213,228`), the rejection is unhandled, so Node (default `--unhandled-rejections=throw`) converts it to an uncaught exception. Exit code is still 1 so CI is unaffected; the damage is purely the confusing, crash-looking output on every failed build (dev mode is fine, since route errors there surface as a clean per-request 500 overlay). Fix: wrap the action bodies (or the sade dispatch) in try/catch that prints the underlying message and `process.exit(1)` without the V8 stack. Not covered by the existing CLI dx entries (`--skip-bundle` strictness, port-in-use, typegen, unknown-types cascade).

`packages/run/src/adapter/index.ts` › `adapter` | 2026-07-19 | impact:med | effort:low

An adapter-less `@marko/run` build serves via `default-entry` -> `createMiddleware`, which builds `platform = createPlatform({ request: req, response: res })` (`adapter/middleware.ts:150-153`), so at runtime `ctx.platform` is a populated node `{ request, response }` (`NodePlatformInfo`). ## Add a client-asset-dir override (or a first-class Cloudflare adapter) so Workers deploys don't fs-rename the build tree

`packages/run/src/vite/plugin.ts` › `CLIENT_OUT_DIR` | 2026-07-19 | impact:med | effort:med

There is no `@marko/run-adapter-cloudflare` package (`packages/adapters/` has only node, static, netlify), so deploying to Cloudflare Workers, the target the maintainers' own marko-idle uses, requires a hand-written adapter plus `wrangler.jsonc`, `worker.ts`, and `env.d.ts`. The core friction is that `@marko/run` hardcodes the client build under `public/`: `const CLIENT_OUT_DIR = "public"` (`plugin.ts:66`) and `if (!isSSRBuild) outDir = path.join(outDir, CLIENT_OUT_DIR)` (`plugin.ts:472-475`). `@cloudflare/vite-plugin` emits `dist/ssr/wrangler.json` with `assets.directory: "../client"`, but marko/run writes the client pass to `dist/public/client`, so the emitted config points at a non-existent dir. The community adapter works around this with a `buildEnd` hook that `fs.renameSync`s `dist/public/client` -> `dist/client` and removes `dist/public`, guarded by brittle path-sniffing (`out.endsWith(path.join("public","client"))`). Nothing in the `Adapter` interface (`packages/run/src/vite/types.ts:35-68`) lets an adapter override just the client asset dir while keeping marko/run's client/ssr split, and `viteConfig`'s single `build.outDir` cannot express it. Fix: expose a per-adapter client-output-dir hook (Cloudflare Workers Assets expects a sibling `client/`, not a nested `public/`), and ideally ship an official Cloudflare adapter. Positive corollary worth noting: the cloudflare SSR bundle is Node-builtin-free (empty `compatibility_flags`, no `nodejs_compat`), and `wrangler dev` served streaming `<await>`, dotted catch-alls, and urlencoded+multipart POST correctly, so the runtime itself is edge/workerd-compatible. (@cloudflare/vite-plugin 1.44.0.)

## Reorder the Netlify edge `ssr.resolve.conditions` so `node` doesn't outrank web-standard conditions

`packages/adapters/netlify/src/index.ts` › `viteConfig` | 2026-07-19 | impact:low | effort:low

For a `netlifyAdapter({ edge: true })` SSR build the adapter sets `ssr.target: "webworker"` with `ssr.resolve.conditions = ["worker", "node", "import", "require", "production", "default"]` (`packages/adapters/netlify/src/index.ts:35-44`), placing `node` in position 2, ahead of `import`/`default`. For a dependency whose package `exports` distinguish `node` vs `browser`/`default` but declare no `worker` condition, resolution then picks the `node` branch. Netlify Edge Functions run on Deno, which only partially polyfills Node, so a node-targeted entry reaching for `Buffer`/`process`/`node:`-only APIs can be bundled into the edge worker and fail at runtime, and an app cannot reorder the array without overriding the adapter's `viteConfig`. Marko's own runtime is web-standard so the shipped starter is unaffected (no runtime failure was observed), which keeps this a latent footgun rather than a live bug. If `node` is a deliberate Deno-compat fallback, placing it after the web-standard conditions (`worker`, then `import`/`browser`/`default`) would keep the fallback while preferring web-standard builds.

## Make `+handler`/`+middleware` unit-testable: expose a way to install the ambient `Run` global and a Context/next test helper

`packages/run/src/runtime/internal.ts` › `globalThis.Run` | 2026-07-19 | impact:high | effort:med

Importing a bare `+handler.ts` in a test fails to even load with `ReferenceError: Run is not defined`, thrown at module-eval of the documented `export const GET = Run.GET(...)`. `Run` is installed only as a side effect of `globalThis.Run ??= {...}` in the non-exported `internal.ts:48`, which a real app pulls in solely through the codegen'd virtual entry `import { ... } from "virtual:marko-run/runtime/internal"` (`codegen/index.ts:361,764`); the public entry does not import it, and `packages/run/package.json` `exports` expose only `.`/`./namespace`/`./router`/`./vite`/`./adapter`/`./adapter/middleware` (no `./internal`, no `./dist/*`), so nothing that installs `Run` is reachable and `import("@marko/run/dist/runtime/internal.js")` throws `ERR_PACKAGE_PATH_NOT_EXPORTED`. `packages/run/README.md` › `+layout.marko` | 2026-07-18 | impact:med | effort:med
