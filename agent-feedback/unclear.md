# Unclear Code & Docs

Things that were hard to understand, and what would have clarified them. Format and rules: [README.md](README.md).

## Document returning a streaming Response / SSE from a +handler, including the disconnect-cleanup caveat

`packages/run/src/runtime/types.ts` › `HandlerReturnValue` | 2026-07-18 | impact:low | effort:low

Returning a streaming `Response` from a `+handler` is a first-class, fully supported capability but is entirely undocumented. The handler return type accepts any `Response` (`HandlerReturnValue = … | Response | Typed<Response, any>`, types.ts:715-722) and the node adapter streams `response.body` chunk-by-chunk (packages/run/src/adapter/middleware.ts:164-169), so a `ReadableStream` body with `content-type: text/event-stream` works. Yet the run docs contain zero mentions of `text/event-stream`, `EventSource`, `ReadableStream`, or Server-Sent Events (grep of website/docs/marko-run/\*.md returns nothing); handler docs only ever show `return new Response("…string…", { status })` (website/docs/marko-run/file-based-routing.md:46,70). The `<await>` "streaming" docs are about progressive HTML/template streaming, not handler-level response streaming, and can mislead a reader searching for "streaming". A short docs section should show a handler returning a streaming SSE `Response`, note that the body flushes incrementally through the node adapter, and document the disconnect story — currently `ctx.request.signal` does not abort on client disconnect and idle streams are not cancelled promptly (see the companion bug at middleware.ts:142). The docs themselves live in the website repo (website/docs/marko-run/file-based-routing.md), but the capability and the surprising cleanup behavior are run's.

## Document that repeated query-string params collapse to arrays in `ctx.search` (documented for `form`, omitted for search)

`packages/run/src/runtime/internal.ts` › `searchParamsToObject` | 2026-07-18 | impact:med | effort:low

`searchParamsToObject` (`packages/run/src/runtime/internal.ts:91-102`) collapses repeated keys to arrays and leaves single keys as strings: `if (key in obj) obj[key] = Array.isArray(prev) ? [...prev, value] : [prev, value]; else obj[key] = value;`. The same function serves both `ctx.search` (`internal.ts:218`) and form bodies (`internal.ts:164`), so `ctx.search[key]` is heterogeneously `string | string[]` depending on how many times the key appears: `?page=1&page=2&sort=name` yields `{ page: ["1","2"], sort: "name" }` while `?page=abc` yields `{ page: "abc" }`. validation.md documents this array rule for `form` ("Repeated field names become arrays", line 101) but the "Params and Search" section never mentions it, and the docs' own search example (`page: Number(value.page) || 0`, validation.md:32) silently produces `NaN -> 0` for `?page=1&page=2` because `Number(["1","2"])` is `NaN`. The type system hides the shape in both directions: raw `ctx.search` with no validator is typed `undefined`/`unknown`, and a search validator receives `Validator<Record<string, any>>` (`types.ts:730`), so neither surfaces the array case. Document the repeated-param-to-array behavior in the Params and Search docs (identical to `form`), and ideally type the search-validator input as `Record<string, string | string[]>` so code expecting a string is warned about `?k=a&k=b` and code doing `search.tag.map(...)` is warned about a single `?tag=a`.

## Document route matching precedence (static > dynamic > catch-all)

`packages/run/src/vite/codegen/index.ts` › `createRouteTrie` | 2026-07-18 | impact:low | effort:low

The routing docs describe the four directory kinds (static, pathless, dynamic `$`, catch-all `$$`) but never state which wins when several could match, even though the router resolves it deterministically. `createRouteTrie` (`codegen/index.ts:1003-1039`) files each segment into separate `static`/`dynamic`/`catchAll` slots, and `writeRouterVerb` emits them in a fixed order: static terminals and children first (switch/if blocks at `:605-635` and `:657-692`), then descent into `dynamic` (`:627-635`, `:688-692`), and finally a fall-through `return` for `catchAll` (`:700-708`). So a static segment beats a dynamic `$`, which beats a `$$` catch-all — fully deterministic at codegen time, but discoverable only by reading generated output. The practical consequence is load-bearing and surprising: a top-level `$locale` dynamic shadows a sibling root `$$rest` for every non-empty first segment, leaving the root catch-all reachable only for `/` itself. `website/docs/marko-run/file-based-routing.md` (Path Structure) should state the precedence and call out this shadowing so authors can choose between a static `/en` special case, a `$locale` dynamic, and a `$$rest` fallback with intent.

## Give shared layouts an ergonomic, typed way to read heterogeneous route meta

`packages/run/src/runtime/types.ts` › `ContextForFile` | 2026-07-18 | impact:low | effort:med

The `app` starter ships `${$global.meta?.pageTitle || "Marko App"}` in its root `+layout.marko`, but this stops type-checking as soon as one route under that layout omits `pageTitle`: `mtc` reports `TS2339 Property 'pageTitle' does not exist on type 'Empty | { pageTitle: string; }'`. `?.` only guards a nullish `meta`, not a missing key on the non-null `Empty` branch of the union. The mechanism: for a template served by multiple routes, `ContextForFile` unions the per-route contexts (`packages/run/src/runtime/types.ts:870-897`), and each route's `meta` comes from `RouteForFileDef.meta`, which resolves to the route's `+meta.json` shape when present and to `Empty` otherwise (`types.ts:523-525`). So a layout over one route with `{pageTitle}` and one without gets `{ pageTitle: string } | Empty`. The union is semantically accurate — at runtime `context.meta` is a single route's `route?.meta || {}` per request — so TypeScript is technically correct; the gap is that there is no ergonomic accessor and the starter ships a pattern that reads as defensive but isn't type-safe. Options: widen a multi-route layout's `meta` to a merged all-optional shape (`Partial<UnionToIntersection<...>>`), have the starter model app-wide meta via a declared interface with safe access, and document the intended pattern for reading an optional meta key across sibling routes. Reproduces only once a sibling route omits the key (it type-checks cleanly when every route defines it), and was also seen going stale in the `mtc`/`routes.d.ts` staleness finding. Category is unclear rather than bugs because the type is correct; the fix is ergonomics + docs, and the trigger straddles the create-marko `app` template and run's meta-union typing.

## Document (and give an ergonomic accessor for) verb-specific `$global.data`; the POST-error re-render pattern breaks past a truthiness check

`packages/run/src/runtime/types.ts` › `ContextForFile` | 2026-07-19 | impact:med | effort:med

The docs recommend answering a POST with `next({ issues })` / `next({ errors, values })` and rendering field-level errors and refilled inputs in the page (`data-loading.md:57-89`; `validation.md:128-129` TIP "pass the narrowed data to the page"), but following it past a truthiness check hits a wall. For a template, `ContextForFile` fixes `Verb = "GET" | "POST"` and unions the per-verb `Context` (`types.ts:870-897`); `RouteFileGroupVerb` adds "GET" for any route that has a page even with no GET handler (`types.ts:294-296`), and that page-only GET verb's `data` falls back to `Record<string, unknown>` via `Route["data"]` (`types.ts:290-292`), collapsing sibling POST data to `unknown` (so `mtc` reports `TS18046: '$global.data.nested' is of type 'unknown'`); when a real GET handler exists, its `{}` data becomes a union member and POST-only keys error on it (`TS2339: Property 'nested' does not exist on type '{} | {...}'`). Worse, the union is semantically accurate at runtime: the same `+page.marko` is served for GET where the POST-only data is absent, so an unguarded `${$global.data.nested.a}` CRASHES the GET render with "Cannot read properties of undefined (reading 'a')". There is no documented discriminant (`$global.method` does not narrow `$global.data`) and no ergonomic per-verb accessor. Fix: add a `$global.method`-discriminated read or a per-verb data accessor, plus a `data-loading.md` example that displays a specific field error and refills an input WITH the GET guard, rather than only checking `issues` for truthiness (`data-loading.md:80`). Distinct from the existing unclear.md meta-union entry ("Give shared layouts an ergonomic, typed way to read heterogeneous route meta"), which unions `meta` across sibling ROUTES for a shared layout: this unions `data` across VERBS for one page's POST handler, is triggered by the framework's headline form-error pattern, and adds a runtime GET-render crash.

## Document the Standard Schema `issues` shape and a per-field error-rendering recipe

`packages/run/src/runtime/types.ts` › `Schema` | 2026-07-19 | impact:med | effort:low

Showing a validation error next to the offending field, the single most common form need, has no doc-sanctioned recipe. `validation.md:112-129` and `data-loading.md:57-89` only show `Response.json({ issues })` / `next({ issues })` and a generic "Please correct the highlighted fields", then link to the standard-schema repo. A newcomer building a `{ [field]: message }` map writes `issue.path?.[0]?.key` and gets `TS2339: Property 'key' does not exist on type 'PropertyKey | PathSegment'`, because `Schema<I,O>`'s failure branch carries `StandardSchemaV1.FailureResult["issues"]` (`types.ts:24-26`) whose `path` is typed `readonly (PropertyKey | PathSegment)[] | undefined` and needs a narrow to reach `.key`. The runtime issue is deeply nested (`[{ kind, type, message, path: [{ type: "object", key: "email", value, ... }] }]`) and nothing in the docs shows that shape, that `path` is optional (undefined for whole-body issues), or how to map issues to a `{ field: message }` object. Fix (low effort): add a `validation.md` example iterating `issues` into a `Record<string, string>` via `issue.path?.[0]?.key` with the narrow, rendering it, and noting `path` may be undefined. Coupled to the per-verb `$global.data` finding but independently actionable (even with the union fixed, per-field display still needs this mapping), and not covered by the repeated-params-to-arrays entry (that is about search/form array collapse).

## Document the protected-page pattern: a middleware that short-circuits with a redirect

`packages/run/src/runtime/internal.ts` › `call` | 2026-07-19 | impact:low | effort:low

The canonical auth-guard, a `+middleware.ts` that reads a session and returns `ctx.redirect("/login")` without calling `next()`, works but is shown nowhere. In `call` (`internal.ts:295-367`) a handler/middleware's returned value is awaited into `response` and `return response || next(data)` (`:367`), so returning a truthy `Response` (e.g. from `ctx.redirect`, `internal.ts:252`) short-circuits and `next` never runs. But the middleware docs (`file-based-routing.md:78-102`) show only a pass-through logging wrapper that always calls `next()`, and a grep of `file-based-routing.md` for redirect/protect/guard/auth finds nothing; that a middleware CAN short-circuit is only inferable by combining two separate statements ("expect... the same shapes as handler exports" and "a handler function may return... a WHATWG response"). Since reading a session and gating access is the framework's own headline middleware use case (`data-loading.md`, `typescript.md:30`), add an explicit "Protecting a route" example of a middleware that redirects unauthenticated requests without calling `next()`. Distinct from the dx "add a cookie/session helper" entry, which is about the missing cookie primitive; this is the missing control-flow example. Low impact because the behavior works and is loosely inferable, recorded because the protected-page archetype has zero doc example.

## Document Cloudflare Workers deployment, the maintainers' own reference deploy target

`website/docs/marko-run/adapters.md` | 2026-07-19 | impact:med | effort:low

`website/docs/marko-run/adapters.md` (and all of `docs/marko-run`) has zero matches for "cloudflare", "worker", or "wrangler"; it documents only Node, Static, and Netlify. Yet the flagship example app (marko-idle) deploys to Cloudflare Workers and ships a non-trivial hand-written adapter (implementing `plugins`/`getEntryFile`/`startDev`/`buildEnd`/`startPreview`), a `wrangler.jsonc`, a `worker.ts` entry, and an `env.d.ts` ambient module, none of which a reader could reconstruct from the docs. The router's platform contract is stable and used by marko-idle: `fetch = fromRuntime("fetch")` (`packages/run/src/runtime/router.ts:15`) takes `(request, platform)`, so a worker forwards `{ env, ctx }` as the platform. Docs should show the worker entry, the `@cloudflare/vite-plugin` + `viteEnvironment: { name: "ssr" }` wiring, the `wrangler.jsonc` (`assets` binding, `main`), the client-asset relocation caveat, and the `@cloudflare/workers-types`-vs-`lib.dom` globals collision that marko-idle's `env.d.ts` documents; at minimum, link to the marko-idle setup as the canonical recipe until an official adapter exists. Distinct from the source-level "no client-asset-dir override / no Cloudflare adapter" dx finding: this is the docs gap. Docs live in the website repo.

## Document the environment-variable model: `.env` secrets are unreachable via `process.env` and only `VITE_`-prefixed vars exist (and leak to the client)

`packages/run/src/adapter/index.ts` › `startDev` | 2026-07-19 | impact:high | effort:low

The universal "put `DATABASE_URL` in `.env`, read `process.env.DATABASE_URL` in a `+handler.ts`" pattern silently yields `undefined`, and the natural "fix" of prefixing it `VITE_` leaks the secret to the browser, all undocumented. Vite's default `envPrefix` is `VITE_`, so only `VITE_*` keys land on `import.meta.env` and are inlined into the client build; non-prefixed keys are `undefined` in both SSR and client. @marko/run never auto-loads `.env` into `process.env`: `loadEnv` (`packages/run/src/vite/utils/server.ts:20`) is dotenv's `config({ path })` and runs only when the CLI `--env <file>` flag is passed (`envFile && (await loadEnv(envFile))`, `adapter/index.ts:119`; also `cli/index.ts`, `commands.ts`), and the vite plugin only injects `pluginConfig.define["process.env.NODE_ENV"]` (`plugin.ts:549`), nothing else. Verified against a production `dist/index.mjs`: server-side `process.env.MY_SECRET` and `import.meta.env.MY_SECRET` are `undefined` while `import.meta.env.VITE_PUBLIC_TOKEN` resolves (and is bundled into client JS); a grep of `docs/marko-run/*.md` for `VITE_`/`import.meta.env`/`process.env`/`.env` finds nothing beyond the `--env` CLI row. Fix (low effort): a `docs/marko-run` env page explaining which mechanism reads a server-only secret, which exposes a public var, and an explicit warning that `VITE_`-prefixed vars are bundled into client JS; a stronger fix is a typed public/private env helper as peer meta-frameworks ship. Distinct from every existing entry (none mention env vars, `.env`, `VITE_`, or `process.env`).

## Document the `+500` error-page contract (`input.error`, HTML-only gating, root-layout wrapping) and provide a global error hook

`packages/run/src/vite/codegen/index.ts` › `renderRouter` | 2026-07-19 | impact:med | effort:low

The error object IS passed to the error page, `context.render(page500, { error }, page500ResponseInit)` (`codegen/index.ts:481`), and typed via an emitted augmentation `export interface Input { error: unknown }` (`codegen/index.ts:954-958`), but `file-based-routing.md:164-171` only states WHEN `+500` renders and never mentions `input.error`, so an author cannot discover how to display or log the error. Three further undocumented sharp edges: (1) both `+404`/`+500` render only when `Accept` includes `text/html` (`codegen/index.ts:454,478`), so a JSON/API request that throws gets a bodyless `new Response(null, { status: 500 })` (`throw error` at `:484` falls to the outer fetch catch); (2) there is no global error hook anywhere in run's runtime/adapter (a grep for `handleError`/`onError`/`reportError` returns nothing), so errors reaching a 500 cannot be logged/reported except by hand-wrapping every handler; (3) the `+500` page is wrapped in the same root `+layout.marko` (`[...route.layouts, route.page]`, `codegen/index.ts:63`; special pages get `layouts: [...currentLayouts]`, `builder.ts:222`), so an error thrown BY the root layout re-throws while rendering the error page, yielding a bare 500 with no fallback. Fix: document `+500.marko` reading `input.error`, the `Accept: text/html` gating and the bodyless non-HTML 500, and a warning that the error page shares the root layout (so it must not depend on the same `$global.data` that may have caused the failure); and expose or document a global error-reporting hook. Existing feedback only notes the static-adapter 404-status bug and the `runtime.md:106` one-liner that a 500 status is returned on uncaught error.
